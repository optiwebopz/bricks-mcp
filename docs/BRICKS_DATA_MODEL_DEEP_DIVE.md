## 1. Post Meta Storage

All Bricks content is stored in WordPress post meta using specific meta keys.

### Core Meta Keys

**File:** `sources/bricks/functions.php` (Lines 84-87)

```php
define( 'BRICKS_DB_PAGE_CONTENT',  '_bricks_page_content_2' );
define( 'BRICKS_DB_PAGE_HEADER',   '_bricks_page_header_2' );

define( 'BRICKS_DB_PAGE_SETTINGS', '_bricks_page_settings' );
define( 'BRICKS_DB_EDITOR_MODE',   '_bricks_editor_mode' );
define( 'BRICKS_DB_TEMPLATE_TYPE', '_bricks_template_type' );
```

### Complete Meta Key Reference

| Meta Key | Constant | Purpose |
|----------|----------|---------|
| `_bricks_page_content_2` | `BRICKS_DB_PAGE_CONTENT` | Page/post element data (main content) |
| `_bricks_page_header_2` | `BRICKS_DB_PAGE_HEADER` | Header template content |
| `_bricks_page_footer_2` | `BRICKS_DB_PAGE_FOOTER` | Footer template content |
| `_bricks_page_settings` | `BRICKS_DB_PAGE_SETTINGS` | Page-level settings (scripts, SEO, etc.) |
| `_bricks_editor_mode` | `BRICKS_DB_EDITOR_MODE` | Which editor is active |
| `_bricks_template_type` | `BRICKS_DB_TEMPLATE_TYPE` | Template type classification |
| `_bricks_template_settings` | - | Template-specific configuration |

### Why `_2` Suffix?

Starting with Bricks v1.2, the `_2` suffix was introduced for better data organization. The legacy keys (without suffix) are:

- `_bricks_page_content` (pre v1.2)
- `_bricks_page_header` (pre v1.2)
- `_bricks_page_footer` (pre v1.2)

The code still references old keys for backwards compatibility but v1.2+ uses `_2` versions.

## 2. Element Data Structure

### Complete Element Schema

**File:** `includes/helpers.php` (Lines 3260-3288)

Each element in the Bricks data array contains these keys:

```php
[
    // Required keys
    'id'       => 'puikcj',           // Unique 6-char ID
    'name'     => 'section',          // Element type identifier
    'parent'   => 0,                  // Parent element ID or 0 for root
    'children' => ['vtjutb'],         // Array of direct child element IDs
    'settings' => [],                 // Element configuration (key-value pairs)

    // Optional keys
    'label'    => 'Custom Label',     // User-defined label for organization

    // Component-specific keys (when element is part of a component)
    'cid'              => 'component-id',    // Component definition ID
    'instanceId'       => 'inst-123',        // Unique instance identifier
    'parentComponent'  => 'cid-456',         // Parent component ID (nested)
    'parentInstanceId' => 'parent-inst',     // Parent instance ID
    'rootComponent'    => 'root-cid',        // Root component (deeply nested)
    'rootInstanceId'   => 'root-inst'        // Root instance ID
]
```

### Storage Structure: FLAT ARRAY (Not Nested Tree)

**Critical:** Bricks stores elements as a **flat array**, not a nested tree structure.

```php
// How elements are stored (FLAT):
[
    [
        'id'       => 'puikcj',
        'name'     => 'section',
        'parent'   => 0,              // Root element (no parent)
        'children' => ['vtjutb'],
        'settings' => []
    ],
    [
        'id'       => 'vtjutb',
        'name'     => 'container',
        'parent'   => 'puikcj',       // Child of section
        'children' => ['jjnqht', 'yvldmi'],
        'settings' => []
    ],
    [
        'id'       => 'jjnqht',
        'name'     => 'heading',
        'parent'   => 'vtjutb',       // Child of container
        'children' => [],
        'settings' => ['text' => 'I am a heading 1']
    ],
    [
        'id'       => 'yvldmi',
        'name'     => 'button',
        'parent'   => 'vtjutb',       // Child of container
        'children' => [],
        'settings' => ['text' => 'Click me']
    ]
]
```

### Hierarchy Representation

```
Section (puikcj)
└── Container (vtjutb)
    ├── Heading (jjnqht)
    └── Button (yvldmi)
```

**Key points:**
- `parent` stores the ID of the parent element (string) or `0`/empty for root elements
- `children` array contains direct child IDs (for fast child lookup)
- Both `parent` and `children` are maintained for bidirectional traversal
- Root elements have `parent = 0` or `parent = ''` (empty string)

### Element ID Generation

**File:** `includes/helpers.php` (Lines 1780-1804)

```php
public static function generate_random_id( $echo = true ) {
    $hash = self::generate_hash( md5( uniqid( rand(), true ) ) );

    if ( $echo ) {
        echo $hash;
    }

    return $hash;
}

public static function generate_hash( $string, $length = 6 ) {
    // Generate SHA1 hexadecimal string (40-characters)
    $sha1        = sha1( $string );
    $sha1_length = strlen( $sha1 );
    $hash        = '';

    // Generate random hash by selecting random chars from SHA1
    for ( $i = 0; $i < $length; $i++ ) {
        $hash .= $sha1[ rand( 0, $sha1_length - 1 ) ];
    }

    // Convert to lowercase
    $hash = strtolower( $hash );

    return $hash;
}
```

### ID Generation Process

1. **Create unique value:** `uniqid( rand(), true )` generates unique ID with microseconds
2. **MD5 hash:** Creates 32-character hash of the unique value
3. **SHA1 hash:** Creates 40-character hash of the MD5
4. **Random extraction:** Selects 6 random characters from the SHA1 string
5. **Lowercase:** Converts final string to lowercase

**Result Format:** `[a-z0-9]{6}`

**Examples:** `puikcj`, `vtjutb`, `jjnqht`, `yvldmi`, `cvrpll`, `paayqb`

### Root Elements

Root elements (no parent) can be identified by:
- `parent === 0`
- `parent === ''` (empty string)
- `parent` key not present
- `!empty($element['parent'])` returns `false`

### Initialization Hook Flow

**1. `init` Hook (Priority 0)**

```php

```

The `init_elements()` method:
- Loads base element class (`includes/elements/base.php`)
- Defines array of 127+ native element names
- Applies `bricks/builder/elements` filter for third-party elements
- Fetches element manager data (enabled/disabled status)
- Calls `register_element()` for each element

**2. `wp` Hook (After Post Context Available)**

```php

```

The `load_elements()` method:
- Called after `$wp_query` is set up
- Loops through registered elements
- Calls `load_element()` for each to get full metadata

### Element Registration

**Method:** `register_element()` (Lines 219-247)

```php
public static function register_element( $file, $element_name = '', $element_class_name = '' ) {
    // Step 1: Include the element PHP file
    require_once $file;

    // Step 2: Auto-detect class name if not provided
    if ( ! $element_class_name ) {
        // Convert 'heading' to 'Element_Heading'
        $element_class_name = 'Bricks\\Element_' . ucfirst( $element_name );
    }

    // Step 3: Instantiate to get basic info
    $element_instance = new $element_class_name();

    // Step 4: Register in static array
    self::$elements[ $element_instance->name ] = [
        'class' => $element_class_name,    // "Bricks\\Element_Heading"
        'name'  => $element_instance->name, // "heading"
        'label' => $element_instance->label // "Heading"
    ];
}
```

## 4. Settings Schema Per Element

### Control Definition Structure

**File:** `includes/elements/base.php`

Each control in `$this->controls` array:

```php
$this->controls['controlKey'] = [
    // Basic properties
    'type'        => 'text',              // Control input type
    'label'       => 'Button Text',       // UI label
    'placeholder' => 'Enter text...',     // Input placeholder
    'default'     => 'Click me',          // Default value if not set

    // Selection controls
    'options'     => [                    // For select/radio/checkbox
        'sm' => 'Small',
        'md' => 'Medium',
        'lg' => 'Large'
    ],

    // UI behavior
    'inline'      => true,                // Display inline vs full-width
    'reset'       => true,                // Show reset button
    'hidden'      => false,               // Hide from UI but process

    // Organization
    'tab'         => 'content',           // Panel tab: 'content' or 'style'
    'group'       => '_layout',           // Control group key

    // CSS generation
    'css'         => [...],               // CSS rules (see below)

    // Units

    // Conditional display
    'required'    => ['icon', '!=', ''],  // Show only if condition met
];
```

### Setting Key Format (Composite Keys)

Settings are stored with composite keys for responsive and state variations:

```
{controlKey}                           // Base: "text", "_margin"
{controlKey}:{breakpoint}              // Responsive: "_margin:tablet_portrait"
{controlKey}:{pseudo}                  // State: "_background:hover"
{controlKey}:{breakpoint}:{pseudo}     // Combined: "_padding:mobile:hover"
{controlKey}:variant-{id}              // Component variant (@since 2.2)
```

**Examples:**
```php
$settings = [
    'text'                        => 'Click me',
    '_margin'                     => ['top' => '20px', 'bottom' => '20px'],
    '_margin:tablet_portrait'     => ['top' => '10px', 'bottom' => '10px'],
    '_margin:mobile'              => ['top' => '5px', 'bottom' => '5px'],
    '_background'                 => ['color' => '#3498db'],
    '_background:hover'           => ['color' => '#2980b9'],
    '_padding:tablet_portrait:hover' => ['top' => '15px'],
];
```

### Dynamic Data Resolution

Dynamic data tags in settings are resolved during render:

**Tag Format:** `{tag_name}` or `{tag_name:modifier}`

**Examples:**
```php
$settings = [
    'text' => 'Welcome, {user_name}!',
    'link' => '{post_url}',
    'image' => [
        'useDynamicData' => '{post_featured_image}'
    ]
];
```

**Resolution Process:**

1. Check setting value for `{tag}` format
2. Route to appropriate provider (post, user, site, ACF, etc.)
3. Replace tag with resolved value
4. Return final value

**Files:** `includes/integrations/dynamic-data/providers/`

### HTML Output Structure

```html
<div id="abc123" class="brxe-container" data-brxe="container">
    <h2 id="def456" class="brx-heading brxe-heading" data-brxe="heading">
        Welcome to My Site
    </h2>
    <a id="ghi789" class="brx-button brxe-button" data-brxe="button" href="/about">
        Learn More
    </a>
</div>
```

**Attributes:**
- `id` - Element ID (same as in database)
- `class` - Element class + custom classes + global classes
- `data-brxe` - Element type identifier

## Appendix: Undocumented Behaviors & Edge Cases

### Element ID Collision

The random ID generation can theoretically produce collisions. Bricks doesn't explicitly check for uniqueness when generating IDs - collisions are statistically unlikely given the entropy of `uniqid() + rand() + sha1 random selection`.

### Parent/Children Redundancy

Both `parent` and `children` are maintained. This is intentional for bidirectional traversal efficiency:
- Finding children: Direct array access
- Finding parent: Direct property access
- No tree traversal needed in either direction

### Empty Settings Object

Elements with no customizations may have:
- `'settings' => []` (empty array)
- No `settings` key at all
- Both are valid and handled

### Root Element Detection

Multiple ways to check if element is root:
```php
if ( empty( $element['parent'] ) )           // Most common
if ( $element['parent'] === 0 )              // Numeric zero
if ( $element['parent'] === '' )             // Empty string
if ( ! isset( $element['parent'] ) )         // Key not present
```

### Global Class Reference Format

Elements reference global classes by ID in settings:
```php
$settings['_cssGlobalClasses'] = ['class-id-1', 'class-id-2'];
```

The class definitions are stored separately in `bricks_global_classes` option.

### Component Instance Resolution

When an element is part of a component instance:
1. `cid` points to component definition
2. `instanceId` uniquely identifies this instance
3. Settings may contain `{property:propName}` placeholders
4. Placeholders are resolved from instance `properties` array

### CSS Loading Modes

Two modes controlled by `cssLoading` setting:
- `'inline'` (default): CSS in `<style>` tag in `<head>`
- `'file'`: External CSS file at `/wp-content/uploads/bricks/css/post-{ID}.min.css`

### Toggle Mode Element (Bricks 2.3+)

Element name: `toggle-mode`
Category: `general`
Label: "Toggle - Mode"
Keywords: light, dark, theme, mode, toggle
Scripts: `bricksToggleMode`

**Controls:**

| Key | Type | CSS Property | CSS Selector | Description |
|-----|------|-------------|--------------|-------------|
| `icon` | icon | — | — | Light mode icon (default: built-in sun SVG) |

| `iconSize` | number (units) | `font-size` | `.toggle.light > *` | Light mode icon size |
| `iconDark` | icon | — | — | Dark mode icon (default: built-in moon SVG) |
| `iconDarkColor` | color | `color`, `fill` | `.toggle.dark > *` | Dark mode icon color |
| `iconDarkSize` | number (units) | `font-size` | `.toggle.dark > *` | Dark mode icon size |
| `ariaLabel` | text | — | — | `aria-label` attribute (default: `"Toggle mode"`) |

**Render output:** `<button>` with two `<span>` children (`.toggle.light` and `.toggle.dark`), CSS controls visibility based on current mode. Default icons are inline SVGs from `BRICKS_PATH_ASSETS/svg/frontend/mode-light.svg` and `mode-dark.svg`.
